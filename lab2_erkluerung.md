
## Что вообще происходит в этой лабе

По шаблону вы должны сделать и сравнить **WireGuard** и **OpenVPN**, затем проверить **MTU / fragmentation**, сделать **throughput tests**, измерить **packet overhead**, а ещё показать трафик **снаружи** и **внутри** туннеля через Wireshark. Отдельно в гайде описан **Wireshark SSH remote capture** через `sshdump`, DNAT на gateway и захват с `ens18` или `wg0`.  

## 1. Что такое VPN

VPN — это зашифрованный туннель между двумя узлами.
Смысл такой:

* внутренний пакет создаётся как обычно
* потом он шифруется
* затем помещается внутрь внешнего туннельного пакета
* по сети идёт уже внешний пакет
* на другой стороне пакет расшифровывается

То есть в обычной сети ты видишь не исходный ICMP/TCP пакет, а его **зашифрованную оболочку**. Это и есть главная идея VPN.

## 2. Почему в лабе так важны два интерфейса: physical и tunnel

В вашем шаблоне отчёта прямо есть пункт **Traffic Observation** для:

* **Physical Interface**
* **wg0** или **tun0/tap0** 

Это значит, что преподаватель почти наверняка спросит:

**Что видно на физическом интерфейсе и что видно на VPN-интерфейсе?**

Правильная логика такая:

### На физическом интерфейсе

Виден **внешний трафик туннеля**:

* внешние IP адреса узлов
* UDP-пакеты WireGuard или OpenVPN
* зашифрованный payload

### На VPN-интерфейсе

Виден **внутренний расшифрованный трафик**:

* ICMP
* TCP
* HTTP
* внутренние VPN IP

Гайд по Wireshark прямо говорит, что можно менять интерфейс захвата, например `ens18` для физического интерфейса и `wg0` для WireGuard, чтобы сравнить шифрованный и расшифрованный трафик. 

## 3. Как работает Wireshark remote capture

Вы не запускаете Wireshark на Linux-сервере. Вместо этого Wireshark на вашем компьютере подключается по SSH к PC1 и удалённо запускает `tcpdump`. Гайд прямо описывает именно этот сценарий через `SSH remote capture: sshdump`. 

То есть схема такая:

* Wireshark работает локально
* по SSH соединяется с PC1
* запускает там `tcpdump`
* поток захваченных пакетов передаётся обратно в Wireshark

Из-за этого в гайде нужен фильтр:

```text
not port 22
```

Чтобы SSH-трафик самого захвата не засорял trace. Это тоже прямо указано в гайде. 

## 4. Зачем нужен DNAT на gateway

По гайду SSH-подключение снаружи приходит не прямо на PC1, а через **gateway**. Поэтому на gateway делается **DNAT rule**, которая перенаправляет входящие SSH соединения дальше на PC1:22. Гайд прямо даёт пример правила `iptables -t nat -A PREROUTING ... -j DNAT --to-destination <PC1>:22`. 

Теоретически это значит:

* пакет приходит на gateway
* gateway меняет **destination address**
* пакет уходит дальше на PC1

То есть **DNAT = изменение адреса назначения**.

## 5. Зачем нужен `setcap` для `tcpdump`

В гайде указано:

```bash
sudo setcap cap_net_raw+ep $(which tcpdump)
```

Это делается, чтобы пользователь `csdc` мог захватывать пакеты **без root**. 

Теоретически ответ преподавателю такой:

> `setcap` gives `tcpdump` the capability to capture packets without running the entire program as root.

## 6. Что такое WireGuard

WireGuard — это современный VPN с очень простой конфигурацией:

* у каждого узла есть private/public key
* создаётся интерфейс `wg0`
* каждому peer указываются:

  * public key другого
  * allowed IPs
  * endpoint

Шаблон отчёта просит у вас отдельно указать:

* команды генерации ключей
* команды запуска интерфейса
* конфиг PC1
* конфиг PC2
* наблюдения на physical interface и `wg0` 

То есть преподаватель точно может спросить:

* что такое peer
* зачем нужен public key
* что такое `AllowedIPs`
* что такое `Endpoint`

Коротко:

* **Peer** — удалённая сторона туннеля
* **Public key** — идентификатор удалённой стороны
* **AllowedIPs** — какие адреса нужно отправлять через этого peer
* **Endpoint** — физический адрес и порт удалённого узла

## 7. Почему WireGuard обычно считается лучше по простоте и скорости

В шаблоне есть раздел **Comparison**, где надо сравнить:

* setup complexity
* performance
* packet overhead 

Обычно ожидаемый вывод такой:

* WireGuard проще настраивается
* WireGuard часто быстрее
* у WireGuard обычно меньше overhead

Почему так объясняют на теории:

* конфиг проще
* протокол современнее
* меньше лишней логики
* меньше overhead по сравнению с OpenVPN

Для отчёта этого обычно достаточно.

## 8. Что такое OpenVPN

OpenVPN — тоже VPN, но более старый и более тяжёлый по конфигурации.
В вашем шаблоне надо выбрать:

* `Mode (tun/tap)`
* `Protocol (UDP/TCP)`
* `Authentication (x509/PSK)` 

То есть преподаватель почти наверняка спросит именно эти три пункта.

### tun vs tap

* **tun** — работает на уровне IP, то есть Layer 3
* **tap** — работает на уровне Ethernet frame, то есть Layer 2

Для обычного маршрутизируемого IP-трафика почти всегда берут **tun**.

### UDP vs TCP

Для VPN обычно лучше **UDP**, потому что TCP внутри TCP создаёт лишние retransmission и лишний overhead. Это классическая проблема **TCP-over-TCP**.

### x509 vs PSK

* **x509** — сертификаты
* **PSK** — pre-shared key

Для учебной лабы обычно проще и быстрее брать **PSK**.

## 9. Что такое fragmentation и почему она появляется в VPN

В шаблоне целый раздел посвящён **MTU & Fragmentation**, там надо проверить размеры `500, 1400, 1450, 1500, 1550`, а потом найти точный предел MTU перед outer fragmentation. 

Теория тут такая:

Каждый VPN добавляет свои заголовки:

* внешний IP header
* UDP header
* заголовки самого туннеля
* иногда дополнительные служебные данные

Из-за этого реальный полезный размер внутреннего пакета уменьшается.

Если обычная сеть может передать максимум 1500 байт, то после добавления VPN overhead внутренний пакет такого размера уже может не поместиться. Тогда появляется **fragmentation**.

То есть:

* без VPN пакет проходит
* с VPN тот же payload может уже не проходить
* потому что внешний пакет становится больше

Очень важная фраза из шаблона:

> Ping payload size != MTU. 

Это значит: размер, который ты задаёшь через `ping -s`, — это ещё не полный MTU. Полный размер больше, потому что к payload добавляются заголовки.

## 10. Что такое MTU

MTU — это максимальный размер пакета, который можно передать без fragmentation.

Если packet size больше MTU, тогда либо:

* происходит fragmentation
* либо пакет вообще не проходит, если стоит запрет fragmentation

Именно поэтому в лабе используется `ping -M do -s ...` — чтобы заставить систему **не фрагментировать**, а честно показать предел.

## 11. Что такое throughput

В шаблоне есть раздел **Throughput Tests**, где надо сравнить:

* No VPN
* WireGuard
* OpenVPN
* по packet size
* latency
* TCP handshakes 

Throughput — это фактическая скорость передачи данных.
Обычно её измеряют через `iperf3`.

Типичный теоретический вывод:

* без VPN throughput выше
* через WireGuard чуть ниже
* через OpenVPN ещё ниже

Причина простая:

* шифрование требует ресурсов
* туннель добавляет overhead
* OpenVPN обычно тяжелее по обработке

## 12. Что такое packet overhead

В шаблоне есть отдельный раздел **Packet Size Overhead**, где нужно сравнить:

* Plain Traffic
* WireGuard
* OpenVPN
* Payload Size
* Tunnel Packet Size
* Overhead 

Теория:

* **Plain traffic** — обычный пакет без VPN
* **Tunnel packet size** — внешний пакет уже после encapsulation
* **Overhead** = насколько туннельный пакет больше обычного

То есть VPN делает пакет **больше**, потому что добавляет служебные заголовки.

## 13. Самые вероятные вопросы от преподавателя

Вот короткие правильные ответы.

### Что видно на physical interface?

Видно **внешний зашифрованный туннельный трафик**.

### Что видно на `wg0` или `tun0`?

Виден **внутренний расшифрованный трафик**.

### Почему нужен `not port 22`?

Чтобы SSH-трафик самого remote capture не попадал в trace. Это прямо указано в гайде. 

### Что делает DNAT?

Меняет **destination address** и перенаправляет соединение дальше.

### Почему нужен `setcap` для `tcpdump`?

Чтобы пользователь мог захватывать пакеты без полного root.

### Почему выбрали `tun`, а не `tap`?

Потому что нужен обычный IP-туннель, а не передача Ethernet frames.

### Почему выбрали `udp`, а не `tcp`?

Чтобы избежать TCP-over-TCP overhead.

### Почему WireGuard обычно проще?

Потому что у него меньше и понятнее конфигурация.

### Почему MTU уменьшается в VPN?

Потому что VPN добавляет encapsulation overhead.

### Почему fragmentation с VPN начинается раньше?

Потому что внешний пакет становится больше, чем без VPN.

## 14. Самая короткая логика всей лабы

Если совсем коротко запомнить суть, то вот она:

* Wireshark remote capture нужен, чтобы снимать трафик с PC1 без GUI. 
* На physical interface виден **зашифрованный внешний трафик**.
* На `wg0` / `tun0` виден **внутренний расшифрованный трафик**. 
* WireGuard и OpenVPN надо сравнить по **сложности настройки, производительности и overhead**. 
* MTU и fragmentation нужны, чтобы показать, что туннель уменьшает эффективный размер полезной нагрузки. 
* Throughput нужен, чтобы показать, что VPN влияет на скорость.
